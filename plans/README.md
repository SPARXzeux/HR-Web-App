## Recon summary

- **Stack**: Next.js (App Router), Tailwind v4 (CSS-first config, `@theme inline` in
  `src/app/globals.css`, no `tailwind.config.*`). No motion library (no Framer
  Motion/Motion, no React Spring, no GSAP, no Radix/Base UI).
- **Existing tokens** (`src/app/globals.css`):
  - `--ease-out-snappy: cubic-bezier(0.23, 1, 0.32, 1)`
  - `--ease-in-out-smooth: cubic-bezier(0.77, 0, 0.175, 1)`
  - `.stagger-item` / `@keyframes stagger-in` — mount-once list entrance, **the only
    place in the codebase with a `prefers-reduced-motion` override**.
  - `@keyframes enter-modal` — modal-open animation (see plan 002).
  - Global `button:active { transform: scale(0.97) }` — press feedback is already
    handled app-wide at the correct frequency tier. Not touched by these plans.
- **Personality**: dense internal ops dashboard (HR/payroll/logistics). Existing
  restraint (subtle stagger, no bounce/confetti) is correct and should be preserved
  — these plans fix broken/missing motion, they don't add flourish.
- **Critical recon finding**: `tailwindcss-animate` (the plugin that gives
  `animate-in`/`fade-in`/`slide-in-from-*` classes meaning) is **not installed** —
  confirmed absent from `package.json` and no `@plugin` directive in `globals.css`.
  Every className using these tokens compiles to nothing. This affects 7 files (see
  plan 001).

## Execution notes (post-implementation)

All 6 plans have been executed directly against source (2026-07-19). A few
notes for whoever reviews the diff:

- **Plan 004 scope gap**: a post-execution repo-wide grep for `transition-all`
  turned up 37 files still containing it — far more than the 10 locations
  plan 004 documented. This is because the original audit's grep for this
  pattern was truncated (output too large) and only the visible preview was
  used as evidence. The 10 originally-identified locations (TopNav.tsx x2,
  Sidebar.tsx x6, OrgCalendar.tsx x1, app/page.tsx x1) have all been fixed
  exactly as planned. The remaining 37 files were deliberately left untouched
  — expanding scope beyond what was vetted in the plan would mean making
  unreviewed changes. If broader `transition-all` cleanup is wanted, that
  should be a new, separate audit/plan (re-run `improve-animations
  performance` or `reconcile` to catch the full set).
- **Plan 002's exit keyframe** ended up using a pure `scale`/`opacity` mirror
  of `enter-modal` (no `translateY`) once the actual entrance keyframe was
  read — `enter-modal`'s `from` state only had `opacity: 0; transform:
  scale(0.95)`, no vertical offset, so the plan's placeholder `translateY(8px)`
  guess was corrected to match reality exactly.
- All CSS additions (plans 001, 002, 006) live in `src/app/globals.css`. All
  reduced-motion coverage added by plans 001, 003, and 006 lives alongside
  each new utility class's definition.

## Not planned (rejected or out of scope)

- **HR Leaves kanban drag-and-drop** (`src/app/(dashboard)/hr/leaves/page.tsx`) —
  native HTML5 `draggable`/`onDragStart`/`onDrop`. A spring-based snap or
  velocity-based drop would require replacing native DnD with a gesture library.
  That's a rewrite, not an animation fix — out of scope for this audit.
- **OrgCalendar month-grid** — considered a mount stagger on the day cells;
  rejected because this is dense data (leave/task conflicts) the user scans to
  answer "am I double-booked," and motion on data-to-be-read hinders more than it
  helps.
- **`AvatarCropperModal.tsx:114` `transformOrigin: 'center center'`** — checked,
  this is correct and deliberate (image pan/zoom/rotate crop tool). Not a finding.
- **Global `button:active` press feedback** — already correctly implemented
  app-wide in `globals.css`. Nothing to fix.
